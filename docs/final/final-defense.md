Group 5 - Final Technical Defense Snapshot

1. The design aims to create a simple, manageable flight aggregation platform within a semester. It uses the MERN stack and JavaScript throughout to reduce complexity and improve collaboration. MongoDB supports flexible, evolving flight data. A backend-focused architecture handles search, data processing, and integration with external APIs, while the frontend stays lightweight. The system focuses on flight planning only, excluding booking and payments to keep scope realistic, emphasizing scalability and clear separation of concerns.

2. The team chose to use the Flight Scraper Sky API instead of building a custom solution or partnering with airlines, making development easier and avoiding data scraping challenges. The trade-off is dependence on API limits and potentially outdated data, but these risks are considered more manageable than the alternatives.

Also, the team chose MongoDB over Firebase to support more complex and flexible queries needed for flight recommendations. This enables advanced filtering, nested data handling, and easier evolution of the data model. The trade-off is increased setup complexity, a higher learning curve, and more responsibility for managing performance and data consistency, but these are considered acceptable given the application’s need for powerful querying.

3. The system's biggest current weakness is reliance on an external API. Every other stack choice we made was well informed with scalability in mind, so the external API is the most likely future bottleneck in our development. Sudden loss of access to this API will cause a good amount of changes throughout the whole system, such as the requests we make, the format data gets cleaned from, and the tests that rely on a stable external request.

4. Our testing and CI pipeline strongly assure us on the strength of this release. Every major communication between our system has tests to verify. Throughout the semester, our tests have repeatedly shone light on problems in the system when we try to push a change. With our application being containerized, dependencies changing between systems was never a problem, which only leaves the system itself which has testing around every corner. There are tests verifying the backend can communicate with the database, tests verifying the frontend can communicate with the backend,tests verifying a user can enter values and a workflow utilizing all of the major components our of system initiates to return data to that user, and several more.

5. The first issue to tackle would be to figure out a bug that we recently discovered. When a user is searching a flight, sometimes a message saying "No Flights Found" will appear. This is false, because when you press refresh, flights populate the page without having to press submit again. We have dived into this problem, but failed to find the reason with our limited resources/time.
